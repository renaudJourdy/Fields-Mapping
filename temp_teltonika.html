Connecting Teltonika device to your flespi channel for the first time

Step-by-step guide on how-to obtain telematics data from Teltonika devices via API.


When you configured device with address of your flespi channel do not forget to reboot the device in order to connect to new server(your flepsi channel). You can do this be sending it "cpureset" command. If you registered device with phone number you can do this via SMS command directly from flespi: Commands & Settings => Actions => Reboot device.

If you have configured the device correctly and rebooted, but it does not connect to flespi, note the following: default device configuration is to store/send records "after time sync". Time can be synchronized using GNSS signal or NTP server. So you may do one of the following:

  1. make sure that device is placed under clear sky
  2. configure valid NTP server
  3. configure records saving/sending without TS to "Always"

Our recommendations for configuration of your Teltonika device

To register received message parameters under correct name with normalized value flespi maps specific AVL ID to corresponding message parameters based on the configured type of device. For different device types this mapping can be different. That's why it is important to select correct device type when you register the device in your flespi account.

If you do not receive from device all expected parameters, check that you configured it to use Codec 8 Extended for data transmission. It can be checked via USB configuring tool or via flespi settings: Commands & Settings => System Params => Data codec.

When device is configured to use plain Codec 8 (not extended) it can not send to flespi any parameter which AVL ID value is greater then 255. Even if you enable such parameter in configuration the device will silently cut it from the message. That's why we always recommend to use Codec 8 Extended for data transmission to prevent any data loss.

You can check Codec 8 configuration in device settings: data_codec setting with codec value 1 indicates that Codec 8 Extended is active, while value 0 indicates that basic Codec 8 is configured. Also in each device message flespi automatically adds codec.id parameter to indicate particular codec device used for transmission of that message. It will have value 8 for simple Codec 8 and 142 (0x8E) for Codec 8 Extended. Looking at this parameter in device telemetry you can understand which data codec device was using for the last data transmission.

To keep device connected to your channel and ready for instant commands over the air we recommend to configure device with extra Connectivity options:

  • Network Ping with value 300 (5 minutes) or below
  • Open Link Timeout with value 600 (10 minutes) or below
  • Server Response Timeout with value 30 (30 seconds)

flespi will automatically close TCP connection if no traffic generated in 10 minutes so in that case Network Ping with 5 minutes will help to keep connection (device will send just single byte into connection).

For most of Teltonika devices they will not reconnect to the channel if connection is closed (by other side) and there is no pending record(message) to deliver to the server. The only option to overcome this is to setup regular records generation even if vehicle is stopped. You can also configure this directly from flespi via Commands & Settings => Tracking tab. We suggest to put 1 to saved records to deliver the record as quickly as possible to server and some meaningful Data Recording Timeout, for example with value 300 (5 minutes):

And finally if you need device to be online all the time do not forget to check that Sleep mode is disabled. You can find Sleep mode configuration in "System Params" tab.

Parameters named custom.param.ID in parsed messages indicate AVL IDs not yet integrated into protocol parsing scheme. Please report such parameters to flespi team via Helpbox for proper integration into standardized parameter names.

Channel traffic encryption

You can enable TLS traffic encryption between Teltonika devices and the flespi channel. This should be activated or deactivated simultaneously in the device and channel configuration. But we suggest to activate TLS encryption only after you ensured that device correctly operates in raw TCP mode to simplify any diagnostics.

If you see in your channel logs just connect/disconnect events without any traffic it means that device was able to establish TCP connection but was not able to establish TLS encryption over it. It means device is either using invalid TLS certificate or simply not configured to use encryption. Disable TLS encryption checkbox from the channel configuration to investigate the real raw traffic device is generating for further diagnostics.

DualCam, DashCam and ADAS configuration

To configure Teltonika DualCam, DashCam or ADAS to send photo or video data to your flespi channel you use the same channel address as you used for telematics data reception. Channel will automatically handle telematics and media data streams from your Teltonika DualCam, DashCam or ADAS camera equipped device.

All video functions are supported for these cameras except live video streaming. However be aware that camera is communicating with Teltonika device via serial cable with limited data transfer capabilities. On top of GPRS connection with high latency this may limit the video upload functionality for you to just small 5-10 seconds clips. During the integration in our tests it could take minutes to deliver even 10 seconds clip with 4G-capable devices and video upload was almost not possible for 2G devices. We suggest to start with photo upload and configure video clips to be as short as possible if will you really need them.

Troubleshooting

If you constantly see the same parsing error in device logs again and again the most popular reason is the corrupted message record stored on the device. And the only option you have is to cleanup storage on the device and wipe out all its messages. You can do this by sending sdformat command to device. You can order flespi to send such command automatically when device connects by activating Commands & Settings => Actions => Format SD card setting.

Tacho functionality

In order to activate tachograph functionality you have to enable "Secondary URI" with type "tacho" in the channel configuration first. This option will provide channel with additional secondary uri to which tracker will send information from tachograph. After you enabled this option further tracker configuration is fully automated by flespi channel and occurs when you execute any tachograph related command.

Tachograph files can only be uploaded using the request_tachograph_file command.

Mandatory conditions for proper operation:

  1. The company's card number in the configuration of the device must match the card number of the connected device Tacho Bridge Card.
  2. The option Secondary URI in the channel should be enabled.
  3. Ignition in the vehicle must be turned ON.
  4. The Tacho Bridge App authorization application must be configured and connected according to the instructions.

Tacho WEB (Automatic file uploading). 

The option is available using the Tacho WEB settings. To configure the flespi server, you must leave the Host field empty and Port equal to 0, the correct configuration will be sent to the device.

The device with the specified timeout will connect to the configured flespi channel. If there is a configured and connected TBA authorization will be completed and the server will automatically request all available files from the tachograph (driver 1-2 and data from memory for 90 days) in total about 350 kb.

With a good connection, downloading ddd files takes about 20 minutes.

Smart ignition check for automatic downloading mechanism.

The check is performed using the telemetry parameter #engine.ignition.status for all device models that use a tacho. This parameter corresponds to AVL_ID=239, which is typically included in the tracker's basic configuration.

Conditions:

  • If the telemetry parameter is present, the download check will only be performed if True.
  • If the telemetry parameter is absent, the check is performed as usual for each incoming message.
  • If the parameter is present but has not been changed recently, the 2-day rule applies. If the value exceeds this limit, the check will be performed as usual for each message.
